Cover Story

把电商客服的复杂意图，设计成可识别、可澄清、可执行的系统。

本展示页包括 6 大意图全景、 ORDER.LOGISTICS.TRACK 完整示例、意图识别 Prompt、单轮与多轮 Query 改写策略、 分层 Fallback 工程方案，以及流程图与泳道图两张资产。

这不是单纯的"分类任务"，而是一条从用户自然表达通向后端动作执行的中间基础设施链路。

为什么展开物流追踪

  • 在电商咨询中占比最高，行业平均 30%+。
  • 涉及多轮对话、槽位提取、状态机跳转，复杂度足够典型。
  • 与退换货、地址修改、投诉等意图天然交叉，最能体现边界设计能力。

系统要解决什么

  • 识别用户真实意图，而不是只看表面关键词。
  • 把口语、短句、代词追问改写成可检索问题。
  • 在置信度不足时主动澄清，而不是强行猜测。
  • 把识别结果映射到 API、RAG 或 SOP 动作。

核心意图与设计重点

核心意图：物流追踪 ORDER.LOGISTICS.TRACK。
设计重点：多轮会话、槽位补全、边界跳转、Fallback。

风格与部署

页面风格：杂志式排版、红色栏目标签、单块章节布局。
部署方式：单文件 HTML，适合 GitHub Pages。

二、完整意图体系设计：物流追踪

选取 ORDER.LOGISTICS.TRACK 作为完整示例，输出意图卡片、层级结构、典型问题、槽位设计、边界定义与动作映射。

2.1 意图卡片
意图 ID

ORDER.LOGISTICS.TRACK

意图名称 / 大类

物流追踪 ｜ 所属物流配送（Logistics）

描述

用户希望了解已下单商品的当前物流状态、包裹位置、预计到达时间，或某个配送节点卡住的原因。

触发场景

下单后、发货后、包裹超时未到。

优先级 / 情绪权重

P0（高频 + 强时效性）｜情绪中-高（超时未到时情绪较激动，需情感安抚）。

改写策略

单轮扩写口语化物流词；多轮消解指代、补全订单实体。触发率预估约 65%。

改写后流向

rewritten_query 替换 original_query 进入 L2 向量检索；原始 query 保留用于日志分析与意图漂移检测。

缓存策略

Redis TTL = 10min，按 (query + last_turn) hash 缓存。

ORDER                                   # L1 大类：订单域

    │   ├── TRACK.ETA_QUERY             # 预计到达时间

  • L2 按用户旅程阶段划分。
  • L3 按具体动作意图划分，粒度到可直接映射后端接口。

2.5 意图边界定义

属于本意图

  • 查询物流当前节点、快递员信息。
  • 询问预计送达日期。
  • 询问包裹滞留 / 异常原因。
  • 查询多包裹拆单情况。

不属于本意图（需跳转）

  • "快递一直不来，我要退货" → ORDER.AFTERSALE.RETURN
  • "快递员态度不好，我要投诉" → ORDER.COMPLAINT
  • "我想改一下收货地址" → ORDER.LOGISTICS.ADDRESS_CHANGE
  • "显示已签收但我没收到" → ORDER.AFTERSALE.MISSING_PACKAGE
  • 模糊地带：若用户先查物流，再说"不行我要退" → 在同一会话中追加 Return 意图，不重置当前意图。
  • 情绪叠加：若 query 同时含物流查询 + 催促情绪 → 识别为 TRACK.STUCK_QUERY，附加情感标签 emotion=anxious。
2.6 对应承接动作（意图识别 → 槽位补全 → 动作路由）
  • sub_intent=STATUS_QUERY + order_id 已知 → 调用 GET /logistics/status?order_id={}
  • sub_intent=ETA_QUERY + 物流已揽收 → 调用 ETA 预测模型 API，返回置信区间时间。
  • sub_intent=STUCK_QUERY + 停滞 > 48h → 触发"物流异常处理 SOP" + 工单创建。
  • order_id 未知 → 触发澄清问答："请问您的订单号是？"
  • 识别置信度 < 0.6 → 触发兜底意图 FALLBACK.CLARIFY。

1. confidence < 0.55 时，intent 必须设为 "FALLBACK.CLARIFY"

4. 订单号格式参考：纯数字 10-20 位；快递单号：字母+数字混合
5. 不要编造信息，未识别到的槽位填 null

Section IV / Query Rewriting

四、Query 改写策略

用户输入 → 预处理 → [Query 改写模块] → 意图识别 → 槽位补全 → Action Router

4.1 为什么要改写
  • 口语化 / 省略："这个多少钱""还有货吗" → 缺乏上下文，向量检索发散。
  • 指代不明："它能退吗""换一个颜色" → 代词无法匹配知识库。
  • 多意图混合："查物流顺便问下怎么退货" → 单次检索覆盖不全。
4.2 单轮改写策略

触发条件：无历史对话，或 query 长度 < 12 字，或含口语化表达。改写目标：将模糊 / 口语化的物流 query 扩展为含标准属性词的完整检索句，提升向量召回率。

#原始 Query改写后策略
01"快递到哪了""我的包裹当前物流位置在哪里，最新运输节点是什么"扩写 + 标准属性词注入
02"东西发货没""订单是否已发货，快递单号是多少，快递公司是哪家"意图拆解扩写
03"什么时候能到""包裹预计到达时间是什么时候，预计几天内送达"ETA 语义扩写

5.2 三层意图识别

L1 规则引擎

正则 + 关键词，优先级最高。超高频 query（如"快递到哪了"）走规则，延迟可控制在 5ms 内。

L2 向量相似度匹配

对比意图标准问题库，召回 Top-3 候选，适合规则无法穷举的中间地带。

L3 LLM 意图分类

当 L1 / L2 置信度不足时调用，延迟可接受在 150-300ms。L1 覆盖 60%，L2 覆盖 25%，LLM 仅处理剩余 15%。

5.3 改写器核心代码
class QueryRewriter:

    def rewrite(self, query: str, history: list[dict]) -> dict:
        """
        Returns:
          rewrite_type: "skip" | "single" | "multi_turn"
          rewritten_query: str
          original_query: str
        """
        rewrite_type = self._judge_type(query, history)

        if rewrite_type == "skip":
            return {"rewrite_type": "skip", "rewritten_query": query}

        if rewrite_type == "multi_turn":
            last_turns = history[-3:]  # 只取最近3轮，控制Token
            rewritten = self._multi_turn_rewrite(query, last_turns)
        else:
            rewritten = self._single_rewrite(query)

        # 改写结果缓存（Redis，TTL=10min）
        cache_key = f"rewrite:{hash(query)}:{hash(str(history[-1:]))}"
        redis.setex(cache_key, 600, rewritten)

        return {
            "rewrite_type": rewrite_type,
            "original_query": query,
            "rewritten_query": rewritten
        }

    def _judge_type(self, query: str, history: list) -> str:
        PRONOUNS = ["它", "那个", "这个", "另一个", "那里", "那边", "那家"]
        ELLIPSIS_PATTERNS = ["呢", "怎么样", "多久", "有吗", "能吗"]

        has_history = len(history) > 0
        has_pronoun = any(p in query for p in PRONOUNS)
        is_short    = len(query) <= 8
        is_followup = any(query.endswith(p) for p in ELLIPSIS_PATTERNS)

        if has_history and (has_pronoun or is_short or is_followup):
            return "multi_turn"
        if is_short or self._is_colloquial(query):
            return "single"
        return "skip"

5.5 / 5.6 / 5.7 质量监控、版本管理与关键 Trade-off
  • 意图准确率：人工抽样标注 / 自动化 A/B，低于 85% 触发迭代。
  • Fallback 率：高于 15% 说明意图覆盖不足。
  • 槽位补全轮数：平均超过 1.5 轮说明槽位设计需优化。
  • 改写跳过率：过低说明 query 质量差，需前端优化。
  • 意图漂移检测：周级新 query 聚类分析，发现新意图簇后扩充意图树。
  • 版本管理：YAML 解耦存储 intent 配置与 Prompt 版本，支持灰度发布与 A/B 测试。
  • Trade-off：只在必要时调 LLM；历史仅取最近 3 轮；改写后再做意图验证；Redis 缓存 TTL = 10 分钟。

T1用户 → 改写模块

用户发起问题："我的快递 3 天没动了，为什么停那么久？"

T2改写模块 → 预处理层

判断为多轮改写场景：既有情绪，也符合停滞追问语义，并且可能存在实体省略。

T3预处理层 → 改写模块

返回清洗后的 query 以及最近 3 轮对话历史，供改写模块拼装完整语义。

T4改写模块 → 意图引擎

改写结果为："物流长时间无更新，停滞原因是什么，停滞多久算异常"。

T5意图引擎 → 槽位管理

识别得到 ORDER.LOGISTICS.TRACK.STUCK_QUERY，并附带 emotion=anxious。

T6槽位管理 → 用户

若缺失 order_id，系统发起澄清追问："请问您的订单号是？"

T7用户 → 槽位管理

用户补充订单号 2024120500123，系统完成核心槽位补全。

T8槽位管理 → Action Router

槽位补全完成，路由到物流异常处理链路。

T9Router → 后端 API / Redis

调用物流状态 API、异常 SOP 或工单创建服务，准备返回结构化处理结果。

T10后端 → 用户

返回结果并写入 Session：current_intent、slots、emotion、last_action，Redis TTL = 30min。
